# Simulated Resilient API

A learning project to practice backend engineering by building a resilient API with real-world distributed systems patterns: replication, caching, queues, artificial latency, intermittent faults, and rate limiting.

## About this project

> **Important:** All the code in this repository was written by me. AI was used **only as a coach/teacher** — guiding me through concepts, explaining the "why" behind decisions, reviewing my code, and pointing out bugs. No code was generated by AI. Every line was typed and decided by me.
>
> This was a deliberate constraint to maximize learning. The goal wasn't to ship a project quickly — it was to internalize the patterns of building resilient backend systems.

## Goals

- Practice **Node.js** and **MongoDB** in a non-trivial setup
- Understand and implement common distributed systems patterns
- Learn how to handle the unhappy path: failures, slow responses, stale data, retries

## Tech stack

- **Monorepo:** npm workspaces
- **Backend:** Node.js, Express, TypeScript, Mongoose
- **Database:** MongoDB (replica set)
- **Cache & Queue:** Redis + BullMQ
- **Frontend:** *(in progress)*
- **Infrastructure:** Docker Compose

## Architecture

```
packages/
  api/        Express + TypeScript API server
  client/     Frontend (in progress)
  shared/     Shared TypeScript types
```

## Simulations implemented

| Simulation | What it does | Where it lives |
|---|---|---|
| **Artificial latency** | Adds random delay to responses | `middleware/latency.ts` |
| **Intermittent faults** | Randomly returns 500s based on a failure rate | `middleware/intermittentFaults.ts` |
| **Rate limiting** | In-memory sliding window per IP | `middleware/rateLimit.ts` |
| **Cache** | Redis-backed read cache with TTL and invalidation | `services/post.service.ts` |
| **Queues** | Async post creation via BullMQ + worker | `queues/` |
| **Replication** | 3-node MongoDB replica set, secondary reads, automatic failover | `docker-compose.yml` |

## Local setup

### 1. Install dependencies

```bash
npm install
```

### 2. Add hostnames to `/etc/hosts`

The replica set members are reachable by Docker service names. Add this line:

```
127.0.0.1 mongo1 mongo2 mongo3
```

### 3. Start infrastructure

```bash
docker compose up -d
```

### 4. Initialize the replica set (first time only)

```bash
docker compose exec mongo1 mongosh --port 27017 --eval '
rs.initiate({
  _id: "rs0",
  members: [
    { _id: 0, host: "mongo1:27017" },
    { _id: 1, host: "mongo2:27018" },
    { _id: 2, host: "mongo3:27019" }
  ]
})
'
```

### 5. Configure environment variables

Copy the sample env file and fill in the values:

```bash
cp packages/api/.env.sample packages/api/.env
```

### 6. Start the API

```bash
npm start --workspace=@resilient/api
```

## Testing the simulations

- Hit `GET /posts` repeatedly to see the cache hit/miss logs
- Spam requests to trigger the rate limiter
- Adjust `failureRate` in `index.ts` to see intermittent 500s
- Run `docker compose stop mongo1` to watch the replica set fail over
